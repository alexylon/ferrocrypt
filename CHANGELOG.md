# Changelog

All notable changes to FerroCrypt are documented in this file.

## [Unreleased]

### Added
- **Bech32 recipient strings (`fcr1...`)** for human-readable public key exchange. The Bech32 payload carries a typed body `type_name_len(2) || key_material_len(4) || type_name || key_material` together with an internal SHA3-256 checksum domain-separated from the BIP 173 checksum, so cross-algorithm collisions are impossible and a copy-paste corruption is caught at decode. Library: build a `PublicKey` via `PublicKey::from_recipient_string("fcr1…")` (or `"fcr1…".parse::<PublicKey>()`) and render one back out with `PublicKey::to_recipient_string()`; `decode_recipient` is kept as the low-level Bech32 primitive that yields raw 32-byte X25519 material. CLI: new `recipient` subcommand (alias `rc`) and `hybrid --recipient` / `-r` flag to encrypt directly with a recipient string instead of a key file.
- **Public-key fingerprints (SHA3-256)** for out-of-band verification, domain-separated by recipient `type_name` so future native types (post-quantum, hybrid KEMs) cannot collide with X25519 fingerprints. CLI: new `fingerprint` subcommand (alias `fp`); `keygen` and `hybrid` encrypt now print the relevant fingerprint (falling back to the key file path if the key cannot be read). Desktop app shows the fingerprint with a copy-to-clipboard button.
- **`--max-kdf-memory <MiB>`** flag on `symmetric` and `hybrid` to cap accepted KDF memory cost during decryption. Library: attach a `KdfLimit` to `SymmetricDecryptConfig` / `HybridDecryptConfig` via the `.kdf_limit(limit)` builder; out-of-budget files fail with `CryptoError::KdfResourceCapExceeded { mem_cost_kib, local_cap_kib }`. `KdfLimit` is `#[non_exhaustive]` so future limit dimensions (time cost, parallelism) can be added without a breaking change — construct via `KdfLimit::new` or `KdfLimit::from_mib`. Default ceiling is **1 GiB**, matching the writer's own default; callers who encrypt with higher `mem_cost` must opt into a wider limit explicitly.
- **`--save-as` / `-s`** flag to choose the encrypted output filename. `--output-path` is optional when `--save-as` is given.
- **Early conflict detection** for `encrypt` and `keygen` — fails before overwriting an existing `.fcr` file or key pair.
- **Subcommand help** with descriptions, aliases, and usage examples. Aliases: `sym` for `symmetric`, `hyb` for `hybrid`, `gen` for `keygen`, `rc` for `recipient`, `fp` for `fingerprint`.
- **Slint-based desktop app** (`ferrocrypt-desktop`) with Symmetric and Hybrid tabs. Key generation is inline within the Hybrid tab; after generation the app auto-transitions to Hybrid Encrypt with the public key pre-filled. Includes "Save As" dialog, magic-byte mode detection, conflict warnings, key file validation on selection, and a password strength indicator (scoring adapted from Proton Pass).
- Progress messages (`Deriving key…`, `Encrypting…`, etc.) printed to stderr.
- `cargo-audit` dependency vulnerability scanning in GitHub Actions.
- `cargo-vet` supply-chain attestation enforced in GitHub Actions.
- **Desktop crate in CI.** A new `desktop` matrix job in `.github/workflows/rust.yml` runs on `ubuntu-latest`, `macos-latest`, and `windows-latest` on every push and pull request. On Ubuntu it installs the Slint system dependencies (`libfontconfig-dev`, `libfreetype-dev`, `libxcb-shape0-dev`, `libxcb-xfixes0-dev`, `libxkbcommon-dev`, `libwayland-dev`, `libssl-dev`), then runs `cargo clippy --all-targets -- -D warnings`, `cargo build`, and `cargo test -- --test-threads=1` inside `ferrocrypt-desktop/`. Previously the desktop crate was excluded from the workspace and never checked in CI, so a library refactor could silently break the GUI until someone ran the build locally. The audit for this job also caught that `README.md`'s Linux install instructions were missing `libwayland-dev` — `rfd 0.17` defaults to the Wayland + XDG-Portal backend and transitively runs a `pkg_config::probe("wayland-client")` at build time, which needs the dev headers. The package was added to both `README.md` and the Fedora install list (as `wayland-devel`).
- **Fuzz-target rewrite around parser surfaces, plus CI wiring.** Seven parser-surface targets exercise the lowest-useful parsing layers directly, skipping the Argon2id derivation that dominates the integration targets: `fuzz_header_prefix` (v1 12-byte prefix + `header_fixed` + recipient-entry framing), `fuzz_private_key_header` (v1 90-byte `private.key` cleartext header + total-size shape check), `fuzz_kdf_params` (12-byte KDF parameter bounds), `fuzz_tlv` (canonical TLV grammar + reserved/critical tag rejection), `fuzz_archive_path` (tar-entry path guard), `fuzz_recipient_decode` (Bech32 `fcr1…` + internal SHA3-256 checksum), and `fuzz_detect_mode` (top-level encryption-mode detection). Two integration targets (`fuzz_symmetric_decrypt`, `fuzz_hybrid_decrypt`) cover the end-to-end decrypt pipeline. A new `fuzz` job in GitHub Actions installs the nightly toolchain and `cargo-fuzz`, runs clippy on the fuzz crate, builds every target on every push/PR (catching API drift), smoke-fuzzes every parser-surface target for ten seconds, and uploads crash reproducers as a workflow artifact on failure. Internals are re-exported through a `ferrocrypt/fuzzing` cargo feature that is off by default and not part of the stable API.
- Library: `PUBLIC_KEY_FILENAME` and `PRIVATE_KEY_FILENAME` constants exported.

### Changed
- **Breaking:** Hybrid encryption migrated from RSA-4096 (OpenSSL) to **X25519 ECDH + HKDF-SHA3-256 + XChaCha20-Poly1305** envelope encryption. Removes the OpenSSL C dependency — the project is now pure Rust. All-zero shared secrets are rejected as a small-order key defence. Pre-`0.3.0` hybrid files cannot be decrypted; old key pairs must be regenerated.
- **Breaking:** New on-disk **format v1** across every artefact (single `.fcr` extension replacing `.fcs` / `.fch`). 4-byte ASCII magic `"FCR\0"` and a 12-byte flat prefix `magic(4) || version(1) || kind(1) || prefix_flags(2) || header_len(4)`. The `kind` byte is `0x45 'E'` for any encrypted file (no per-mode `'S'` / `'H'` byte): the symmetric-vs-hybrid distinction is derived from the recipient list, not encoded in the prefix. `private.key` uses `kind = 0x4B 'K'`. See `ferrocrypt-lib/FORMAT.md` for the full spec.
- **Breaking:** v1 `.fcr` is a single container with a typed **recipient list**, not a per-mode envelope. Wire layout: `prefix(12) || header_fixed(31) || recipient_entries(N) || ext_bytes || header_mac(32) || payload`. `header_fixed` is `header_flags(2) || recipient_count(2) || recipient_entries_len(4) || ext_len(4) || stream_nonce(19)`; each recipient entry is `type_name_len(2) || recipient_flags(2) || body_len(4) || type_name || body`. v1.0 ships two native recipient types: `argon2id` (passphrase, 116-byte body, `MixingPolicy::Exclusive`) and `x25519` (public-key, 104-byte body, `MixingPolicy::PublicKeyMixable`). Mode classification: exactly one `argon2id` entry → `Symmetric`; one or more supported `x25519` entries (no `argon2id`) → `Hybrid`; mixing rejected as `PassphraseRecipientMixed` before any KDF runs.
- **Breaking:** Both modes use a per-file **32-byte `file_key`** indirection (age-inspired): the recipient body seals the random `file_key` via XChaCha20-Poly1305 (passphrase → Argon2id → HKDF-SHA3-256 for `argon2id`; X25519 ECDH → HKDF-SHA3-256 for `x25519`). The payload AEAD key and the header MAC key are derived from `file_key` via HKDF-SHA3-256. Payload encryption is XChaCha20-Poly1305 STREAM-BE32 over 64 KiB chunks; HMAC-SHA3-256 authenticates `prefix(12) || header_fixed || recipient_entries || ext_bytes` and is verified **after** a recipient successfully unwraps a candidate `file_key` (per FORMAT.md §3.7, MAC verification is the final acceptance gate). On a multi-recipient file a per-candidate MAC failure surfaces as `HeaderMacFailedAfterUnwrap { type_name }` and the loop continues to the next supported entry; a single-recipient MAC failure surfaces as `HeaderTampered`.
- **Breaking:** STREAM-BE32 conformance per FORMAT.md §5: writers MUST NOT append an extra empty final chunk after non-empty plaintext whose length is a multiple of 65,536 bytes. The encrypt path defers committing a full chunk until either more plaintext arrives (then it goes out as `next`) or `finish()` is called (then it goes out as `last`); the decrypt path uses a one-byte peek past each `ENCRYPTED_CHUNK_SIZE` boundary to distinguish "exact-N final chunk" from "exact-N then more data". AEAD-BE32's per-chunk `last_flag` binding rejects any chunk-boundary truncation as a tamper failure.
- **Breaking:** `public.key` is a **UTF-8 text file** containing the canonical Bech32 `fcr1…` recipient string optionally followed by a single trailing LF (no other surrounding whitespace). The Bech32 payload carries a typed body `type_name_len(2) || key_material_len(4) || type_name || key_material` plus an internal SHA3-256 checksum (in addition to the BIP 173 checksum). For X25519 the file is ~107 bytes on disk.
- **Breaking:** `private.key` v1 is passphrase-protected via **Argon2id + HKDF-SHA3-256 + XChaCha20-Poly1305** and written with `0o600` on POSIX. The cleartext fixed header is **90 bytes**: `magic(4) || version(1) || kind(1=0x4B 'K') || key_flags(2) || type_name_len(2) || public_len(4) || ext_len(4) || wrapped_secret_len(4) || argon2_salt(32) || kdf_params(12) || wrap_nonce(24)`. For the v1 X25519 case (`ext_len = 0`) the total file is **176 bytes**: header(90) + `"x25519"`(6) + public_material(32) + ext(0) + wrapped_secret(48). Every cleartext byte before `wrapped_secret` is bound as **AEAD associated data** so tampering any field fails authentication. The AEAD primitive returns a single undifferentiated failure for both wrong-passphrase and tampered-cleartext — both surface as `KeyFileUnlockFailed` with Display wording `"Private key unlock failed: wrong passphrase or tampered file"`. The kind byte `0x4B` is deliberately distinct from the encrypted-file kind `0x45` so `file(1)`-style matchers can disambiguate.
- **Breaking:** Forward-compatible authenticated **TLV extension region** (`ext_bytes`, empty today with `ext_len = 0`) is present in both `.fcr` and `private.key`. Entries are `tag(u16 BE) || len(u32 BE) || value`. Tags in `0x0001..=0x7FFF` are ignorable (authenticate + skip); `0x8001..=0xFFFF` are critical (reject with `UnknownCriticalTag`); `0x0000` / `0x8000` are reserved. `ext_len` is capped at 64 KiB. Rules are normative in FORMAT.md §6.
- **Breaking:** SHA-3 unified across the spec — HMAC-SHA3-256 (header MAC), HKDF-SHA3-256 (every derivation: argon2id wrap, x25519 wrap, private-key wrap, payload subkey, header subkey), SHA3-256 fingerprint, SHA3-256 internal Bech32 checksum.
- **Breaking:** Public-key fingerprints hash `type_name || key_material` via SHA3-256, domain-separated so future native types cannot collide with X25519 fingerprints.
- **Breaking:** Symmetric password-based key derivation pipeline: **Argon2id** (1 GiB memory / time cost 4 / parallelism 4) → **HKDF-SHA3-256** → `wrap_key`. The wrap key seals the random 32-byte `file_key` inside the `argon2id` recipient body, and `payload_key` + `header_key` are subsequently derived from `file_key`. A compromise of one subkey does not reveal the other. KDF parameters are stored in the `argon2id` recipient body so decryption always uses the exact values from encryption time.
- **Breaking (CLI):** Removed `-p` / `--passphrase` flag. Passphrases are prompted interactively with hidden input and confirmation on encrypt and keygen; passphrase confirmation uses constant-time comparison. For non-interactive use (scripts, CI) set the `FERROCRYPT_PASSPHRASE` environment variable. Empty passphrases are rejected.
- **Breaking (CLI):** Removed `--bit-size` / `-b` and `--large` / `-l` flags (RSA-only).
- **Breaking (library API):** Each operation function now takes a config struct plus an `impl Fn(&ProgressEvent)` callback and returns a typed outcome struct. The five public operations are `symmetric_encrypt` / `symmetric_decrypt` / `hybrid_encrypt` / `hybrid_decrypt` / `generate_key_pair`, paired with `SymmetricEncryptConfig` / `SymmetricDecryptConfig` / `HybridEncryptConfig` / `HybridDecryptConfig` / `KeyGenConfig` inputs and `EncryptOutcome` / `DecryptOutcome` / `KeyGenOutcome` outputs. Every config and outcome type is `#[non_exhaustive]` and `Clone`; configs are built with `::new(required fields)` plus `.save_as(path)` / `.kdf_limit(limit)` builders, so adding new optional fields or outcome fields in later releases is non-breaking. `ProgressEvent` is a `#[non_exhaustive]` enum whose `Display` wording is locked in by a unit test; consumers that rendered the previous `&str` callback via `eprintln!("{msg}")` can keep doing the same with `eprintln!("{ev}")`. Public-key material is carried by a new `PublicKey` type (constructors `from_key_file(path)`, `from_bytes([u8; 32])`, `from_recipient_string("fcr1…")`, plus `FromStr`) with methods `fingerprint()`, `to_recipient_string()`, `to_bytes()`, and `validate()`. Private-key material is carried by `PrivateKey` (constructor `from_key_file(path)`). Both wrappers are `#[non_exhaustive]` so additional key sources (in-memory, hardware-backed) can be added without breaking the config shape. The auto-routing wrappers (`symmetric_auto` / `hybrid_auto`), the separate `hybrid_encrypt_from_recipient` entry point, and the free functions `public_key_fingerprint` / `encode_recipient` / `encode_recipient_from_bytes` are removed — direction detection is now a caller concern via `detect_encryption_mode`, and the public-key operations moved to methods on `PublicKey`. `decode_recipient` remains as the low-level Bech32 primitive.
- **Encrypt vs. decrypt routing uses magic-byte detection only** — file extension no longer forces a path. `detect_encryption_mode` reads the prefix, parses the recipient list, and returns a classification (`Symmetric` / `Hybrid`) without running any cryptographic work. Files that start with the FerroCrypt magic bytes but have a corrupted prefix or recipient list now fail closed with `InvalidFormat` (typed `BadMagic` / `WrongKind` / `MalformedHeader` / `OversizedHeader` / `MalformedRecipientEntry` / etc.) instead of being silently re-encrypted.
- **Atomic output handling.** Encrypted files and generated key files are staged under temporary names and promoted to their final path only on success, which keeps ordinary single-writer use from silently replacing an existing file. Directory decryption continues to use an `.incomplete` working directory that is moved into place once extraction finishes; on failure the `.incomplete` copy remains on disk for inspection. Linux and macOS keep the stronger directory-finalization path. Windows keeps a narrower best-effort directory-finalization step in exchange for the zero-unsafe design. Encrypted output is flushed to disk before being promoted, and the parent directory is synced on Unix on a best-effort basis after finalization. The library contains no unsafe code.
- **File and directory permissions are preserved** through encrypt/decrypt round-trips on Unix. Setuid, setgid, and sticky bits are stripped on both archiving and extraction. Decrypted regular files are no longer marked executable. On non-Unix platforms, permission handling is platform-limited.
- Tauri and Dioxus GUI experiments moved to `experiments/`; replaced by the new Slint desktop app.

### Fixed
- **Decryption errors are now distinct and user-readable.** Previously a failed decryption surfaced as a wrapped internal error such as `Io(Custom { ..., error: "aead::Error" })` on the CLI, with no way for a user to tell whether the password was wrong, the header was tampered with, the payload was corrupted mid-stream, or the file was simply truncated. The library now exposes a typed taxonomy of failure modes:
    - **Private key unlock failed: wrong passphrase or tampered file** (`KeyFileUnlockFailed`) — the `private.key` file failed AEAD-AAD authentication. Either the passphrase does not decrypt it or one of its cleartext fields has been tampered with since the file was written; the AEAD primitive cannot distinguish the two cases.
    - **Decryption failed: recipient `argon2id` unwrap failed** (`RecipientUnwrapFailed { type_name: "argon2id" }`) — wrong passphrase, or the `argon2id` recipient body was tampered with.
    - **Decryption failed: recipient `x25519` unwrap failed** (`RecipientUnwrapFailed { type_name: "x25519" }`) — the supplied private key does not match the encrypted file's `x25519` recipient slot, or the recipient body was tampered with.
    - **Decryption failed: recipient `<type>` MAC mismatch** (`HeaderMacFailedAfterUnwrap { type_name }`) — in a multi-recipient hybrid file, a candidate slot unwrapped a `file_key` but the resulting `header_key` did not verify the header MAC. The decrypt loop continues to the next supported `x25519` slot. Distinct from `HeaderTampered`, which is the final single-candidate variant.
    - **Decryption failed: header tampered after unlock** (`HeaderTampered`) — for a single-recipient file, the recipient body unwrapped successfully but the MAC over `prefix || header_fixed || recipient_entries || ext_bytes` failed.
    - **Decryption failed: no recipient could unlock the file** (`NoSupportedRecipient`) — the recipient list was iterated to exhaustion without any supported entry yielding a `file_key` that verified the MAC. Also surfaces when `symmetric_decrypt` is invoked on a file whose recipient list has no `argon2id` entry, and vice versa.
    - **Passphrase recipient mixed with another recipient** (`PassphraseRecipientMixed`) — the file declares `argon2id` alongside any other recipient. Per FORMAT.md §3.4, `argon2id` is exclusive; the mix is rejected before any KDF runs so a hostile mixed file cannot force expensive Argon2id work.
    - **Unknown critical recipient: `<type>`. Upgrade FerroCrypt.** (`UnknownCriticalRecipient { type_name }`) — the file declares a `recipient_flags.critical = 1` entry whose `type_name` is unknown to this implementation.
    - **Payload authentication failed: data tampered or corrupted** (`PayloadTampered`) — a ciphertext chunk failed its authentication tag after the header authenticated successfully.
    - **Encrypted file is truncated** (`PayloadTruncated`) — the encrypted stream ends before its final-flag authenticated chunk.
    - **Encrypted file has unexpected trailing data** (`ExtraDataAfterPayload`) — bytes remain after the final-flag chunk. STREAM-BE32's per-chunk nonce binding rejects ordinary appends earlier as `PayloadTampered`; this dedicated variant fires from the post-`decrypt_last` trailing-data probe in `DecryptReader::fill_buffer` for pathological readers that signal end-of-file at the chunk boundary and then yield more bytes.
    - **KDF resource cap exceeded (`required` KiB, cap `cap`)** (`KdfResourceCapExceeded { mem_cost_kib, local_cap_kib }`) — KDF parameters from a header request more memory than the caller's `KdfLimit` allows.

  Encryption and streaming internals no longer leak low-level crate error names into user-visible messages. The CLI prints errors using `Display` instead of the default `Debug` format, so users see the library's short messages directly. The desktop app already used `Display`. The library's error type groups variants by area of concern (filesystem/input, format/version, KDF/work limits, authentication, primitives, internal invariants) and the variants are identity-only — no per-operation context is baked into the error type. Variants that do carry data now carry *typed* data instead of heap-allocated strings: `InvalidFormat(FormatDefect)`, `UnsupportedVersion(UnsupportedVersion)`, `InvalidKdfParams(InvalidKdfParams)`, and `InternalInvariant(&'static str)` / `InternalCryptoFailure(&'static str)`. `FormatDefect` distinguishes structural defects across `.fcr`, `private.key`, and `public.key`: `BadMagic`, `Truncated`, `ExtTooLarge { len: u32 }`, `MalformedTlv`, `UnknownCriticalTag { tag }`, `NotAKeyFile`, `WrongKeyFileType`, `MalformedPublicKey`, `WrongKind { kind }`, `MalformedHeader`, `OversizedHeader { header_len }`, `RecipientCountOutOfRange { count }`, `MalformedTypeName`, `MalformedRecipientEntry`, `RecipientFlagsReserved`, `MalformedPrivateKey`. `FormatDefect`, `UnsupportedVersion`, and `InvalidKdfParams` are re-exported from the crate root so library consumers can pattern-match on structural failure shapes without substring comparisons. The former catch-all `InternalError` variant is split into `InternalInvariant` (state-machine misuse or impossible-size checks that indicate a library bug) and `InternalCryptoFailure` (a crypto primitive rejected well-formed input — practically unreachable, left typed so it cannot be silently reclassified as generic I/O). `CryptoError::InputPath` is now a bare unit variant (was `InputPath(String)`) — the path the caller passed in is information the caller already has, so it does not belong inside the error. The remaining `String`-carrying variant is `InvalidInput(String)`, kept as the designated heterogeneous-caller-input bucket because the tar-archive layer needs to identify *which* archive entry triggered a fail-closed rejection — see the type-level docs on `CryptoError` for the full rationale and the `age` crate parallel.
- **Security: Windows no longer silently overwrites existing output.** Earlier versions on Windows could replace an existing `.fcr` file or key file at the output path without asking. Encryption and key generation now avoid silently replacing existing final files on the supported desktop targets, and directory decryption follows the same intent with a stronger path on Linux/macOS and a narrower best-effort step on Windows.
- **Security: duplicate entries inside a decrypted archive no longer overwrite each other.** A maliciously crafted encrypted archive that contained two entries for the same path previously had the second entry quietly replace the first during extraction. Extraction now fails closed on the duplicate.
- **Security: extraction rejects archives with multiple top-level roots.** FerroCrypt's archiver always produces exactly one top-level root — a single file entry, or one directory entry with descendants. Earlier extraction code tolerated hand-crafted or tampered `.fcr` payloads that contained entries under multiple distinct top-level roots: it would silently create every root under an `.incomplete` working name and rename each into place, while the public decryption return value only reported the first root. A crafted payload could therefore produce more output than the API advertised. Extraction now refuses any second distinct top-level root with an `InvalidInput` error before that root's output is written, so the single `PathBuf` return value is always faithful to what extraction created on disk.
- **Security: key generation writes the public key before the private key.** If generation is interrupted partway, the private key never appears on disk — only an orphan public key remains, which is harmless.
- **Security: directory encryption no longer follows symlinks.** Symlink inputs and symlink entries inside directories are rejected. Encrypt entry points reject symlink and non-regular inputs up-front, before any Argon2id work runs; the archiver re-applies the same per-entry rules during the TAR walk as defense-in-depth. File opens use `O_NOFOLLOW` on Unix with post-open regular-file validation, hardening against symlink and special-file TOCTOU races. Hardlinks are archived as regular file contents. Extraction rejects unsupported TAR entry types instead of silently dropping them.
- **Security: extraction is hardened against local symlink/directory-component races on Linux and macOS.** Every filesystem operation inside the `.incomplete` working tree is now anchored at a directory file descriptor rooted in the user's output directory. Intermediate components are resolved via `openat`/`mkdirat` with `O_NOFOLLOW | O_NONBLOCK`, each resulting fd is `fstat`-verified to be a directory, and the final regular file is created with `O_CREAT | O_EXCL | O_NOFOLLOW`. A concurrent local attacker who swaps a directory component for a symlink no longer redirects writes outside the destination tree: the walker aborts with `ELOOP` or a typed "symlink in extraction path" error. Deferred directory permissions are now applied via `fchmod` on the still-open fd instead of path-based `set_permissions`. Windows and non-Linux/non-macOS Unix targets keep the existing path-based extraction path.
- **Security: KDF parameters from untrusted headers are bounded in every dimension** (`mem_cost`, `time_cost`, `lanes`, plus rejection of zero values), preventing denial-of-service via crafted encrypted files or key files. The local cap is enforced via `KdfLimit` before Argon2id runs; over-budget headers surface as `KdfResourceCapExceeded` rather than triggering the allocation.
- **Security: per-chunk plaintext buffers in the streaming AEAD adapters are zeroized** between chunks, on `finish()`, and on `Drop` — preventing plaintext from lingering on the heap after error or normal completion. Decrypted key material (recipient-derived `file_key`, raw X25519 private key bytes before they enter `StaticSecret`) lives in `Zeroizing` wrappers that clear automatically on drop, covering success, error, and unwinding paths.
- **Desktop: worker thread panics no longer freeze the UI.** Previously, a panic in the encrypt/decrypt/keygen worker (e.g. Argon2id OOM on a constrained host) left the UI permanently disabled with the "working…" state stuck. The worker body is now wrapped in `std::panic::catch_unwind`, resetting `is_working` and surfacing a clear error on panic so the app stays recoverable without a force-quit.
- **Hybrid decrypt progress phases.** `hybrid_decrypt` now emits `DerivingKey` before the private-key Argon2id runs and `Decrypting` only after the recipient body unwraps — matching the symmetric path. Previously it emitted `Decrypting` immediately and never emitted `DerivingKey`, so a UI would mislabel the multi-second KDF window as "decrypting".
- **Pub/priv key file mix-ups are now reported as `WrongKeyFileType`** across every entry point. Reading a binary `private.key` through `read_public_key` used to surface a cryptic `io::Error` "stream did not contain valid UTF-8"; reading a text `public.key` through the X25519 private-key reader or `validate_private_key_file` used to surface the generic `NotAKeyFile` defect. Both paths now detect the swap via `KeyFileKind::classify` (cheap binary signature for `private.key`, Bech32 sniff for `public.key`) and return `FormatDefect::WrongKeyFileType`.
- **`public.key` enforces strict canonical whitespace.** Per FORMAT.md §7.4 the file MUST be the lowercase `fcr1…` recipient string optionally followed by exactly one trailing `\n`. Any other surrounding whitespace — CRLF line endings, leading blanks, trailing spaces, extra blank lines, internal whitespace — is rejected as `MalformedPublicKey` rather than silently trimmed.
- **Missing key-file paths surface as `InputPath`.** `read_public_key`, the X25519 private-key reader, and `validate_private_key_file` now map `io::ErrorKind::NotFound` to `CryptoError::InputPath`, matching the upfront `validate_input_path` behaviour instead of leaking a raw OS error string.
- **`InvalidKdfParams` Display wording.** "File has invalid decrypt settings" → "File has invalid KDF settings": neutral across the encrypt/decrypt split and accurate for both directions.
- Removed the dead `CryptoError::SliceConversion` variant; no code path produced it after the crate's internal `try_into` failure sites all moved to typed `InvalidFormat` / `InternalInvariant` arms. The enum is `#[non_exhaustive]`.
- Output-conflict wording unified to "Output already exists: `path`" across symmetric / hybrid / archive paths (was a mix of "Output already exists" and "Output file already exists"). Symlink-race wording unified to "Input is a symlink: `path`" (was a mix of "Path" and "Input" phrasings for the same class).
- Crash on truncated, corrupted, or maliciously crafted `.fcr` files replaced with structured `InvalidFormat` errors.
- Default encrypted output naming for directories with dots (e.g. `photos.v1/`) preserves the full directory name (`photos.v1.fcr` instead of `photos.fcr`).
- Nonexistent input paths no longer silently produce empty encrypted files.
- `keygen` creates missing output directories. Key pair generation is atomic: both files are written to temp names, `fsync`'d to disk, and renamed into place; partial state is cleaned up on failure.
- Directory archiver no longer silently skips inaccessible files or path errors.

## [0.1.0] - 2023-07-04

### Added
- Symmetric encryption with XChaCha20-Poly1305 + Argon2id key derivation
- Hybrid encryption with RSA-4096 + XChaCha20-Poly1305 (envelope encryption)
- RSA key pair generation with passphrase-protected private keys
- Reed-Solomon error correction on cryptographic headers (salt, nonce, key hash)
- Stream encryption mode for large files (EncryptorBE32/DecryptorBE32)
- Automatic file/directory archiving (ZIP) before encryption
- Constant-time key hash comparison to prevent timing attacks
- Key zeroization after use
- CLI with subcommands: `keygen`, `hybrid`, `symmetric`
- Tauri desktop GUI application
- `.fcr` file format for both symmetric and hybrid encryption
- GitHub Actions CI workflow
